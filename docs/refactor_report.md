# 重构报告：知乎数据监控中心

## 1. 目标与范围
本次重构以「可读性、可维护性、可拓展性」为核心目标，覆盖后端结构、前端资源组织和启动入口，不改变现有 API 协议与业务行为。

## 2. 现状诊断（重构前）
- 后端 `main.py` 约 570 行，混合了启动逻辑、数据库访问、业务服务、定时调度、全部 API 路由。
- 前端 `templates/index.html` 约 1131 行，HTML/CSS/JS 内联耦合，修改一处样式或交互需在超长文件中定位。
- 主要风险
  - 变更风险高：单文件改动影响面大，回归测试难聚焦。
  - 复用困难：抓取逻辑、校验逻辑、统计逻辑无法被独立复用。
  - 扩展成本高：新增 API、替换数据库、引入多页面前端都需要大规模改动。

## 3. 多角色深入讨论（模拟）
### 3.1 架构师视角
结论：采用「分层 + 边界清晰」的最小可行架构，而不是一次性引入过重框架。

- 关注点
  - 路由层只做入参与出参，不承载业务规则。
  - 服务层负责业务流程编排（抓取、校验、调度）。
  - 仓储层封装 SQL，降低 SQL 分散和重复风险。
- 约束
  - 保持 Flask + SQLite 技术栈，避免过度重构。
- 决策
  - 引入 `app/` 包结构，采用 Application Factory + Blueprint。

### 3.2 高级开发工程师视角
结论：优先拆出“高变更频率 + 高认知负担”的模块。

- 关注点
  - 配置校验逻辑与抓取逻辑要单测友好。
  - 路由函数应短小，方便代码评审。
  - 调度器需要可停止、可观测。
- 决策
  - 调度封装为 `SchedulerRunner`，使用 `Event` 控制退出。
  - 文章、统计、配置 API 拆为独立路由模块。

### 3.3 Python 专家视角
结论：在不引入 ORM 的前提下，通过模块化、类型标注和职责隔离获得 Python 可维护性收益。

- 关注点
  - 函数命名与返回结构稳定。
  - 避免全局状态污染（如全局运行标志）。
  - I/O（HTTP/DB）逻辑集中，便于替换实现。
- 决策
  - `repositories` 层统一 DB 访问。
  - `services/zhihu_client.py` 集中封装知乎请求与认证校验。
  - `services/metrics.py` 独立汇总与点击率计算。

### 3.4 资料搜集者视角
结论：采用社区长期稳定实践，避免一次性“架构跃迁”。

- 对标实践
  - Flask 官方常见组织方式：Application Factory + Blueprint。
  - 小型项目演进策略：先“模块分离”，后“契约稳定”，最后再考虑引入 ORM 或任务队列。
  - 前端演进策略：先抽离静态资源（CSS/JS），再逐步组件化。
- 决策
  - 本轮不引入 Celery/Redis，不引入前端构建工具，保持部署成本不变。

## 4. 重构设计
### 4.1 后端分层
- `app/repos.py`：数据库连接、建表初始化、旧配置迁移、SQL 查询与写入。
- `app/services.py`：业务逻辑（知乎请求、认证校验、指标汇总、调度器）。
- `main.py`：Flask 路由与程序启动编排。

### 4.2 前端分层
- `templates/index.html`：页面结构与语义标记。
- `static/css/app.css`：样式。
- `static/js/app.js`：交互逻辑与 API 调用。

### 4.3 稳定性策略
- 保持原 API 路径和返回字段不变。
- 保持默认端口、数据库文件、迁移行为不变。
- 保持用户操作路径不变（保存配置、查看趋势、文章管理）。

## 5. 重构落地结果
### 5.1 新目录结构
```text
zhihu-data/
├── app/
│   ├── repos.py
│   └── services.py
├── static/
│   ├── css/app.css
│   └── js/app.js
├── templates/index.html
└── main.py
```

### 5.2 可维护性收益
- 单文件复杂度显著下降：后端从超长单文件拆为路由入口 + 两个职责模块。
- 模块职责更清晰：SQL 与业务流程分离，减少改动时的连带影响。
- 可拓展性增强：未来新增 API 或切换数据源时，影响面可局部控制。
